港美股日报规则 v1.0 Final SOP

Status: final locked content SOP.

本文件是港美股日报的内容生产与验收规则。它不是开发文档，也不是讨论稿。

区块链日报 SOP 只作为质量标准参考：参考它的模板锁定、信源优先级、置信度、待补追查、生成检查、发布检查和漂移判定方法；不得复用区块链日报的栏目、数据源池和正文内容。

后续所有港美股日报的生成、修改、发布、排查、模板修复，都必须先读取本文件，并按本文件执行。除非用户明确说“修改港美股日报规则”“升级模板版本”“更改底层模板”，否则不得改变本文件规定的栏目结构、字段、数据处理规则、网页样式和发布流程。

  • 生成港美股日报网页。

  • 单只股票深度研报。
  • 项目投资备忘录。
  • 非日报类资讯总结。

区块链内容默认不得进入港美股日报。只有满足以下任一条件时，才允许进入港美股日报：

  • 相关公司已在美股或港股上市。
  • 相关公司已提交 IPO、S-1、F-1、SPAC、港交所聆讯或招股书。
  • 事件涉及美股或港股 ETF、上市基金、交易所审批。
  • 事件影响上市公司财报、并购、融资、监管处罚或股价。
  • 事件影响金融机构、券商、支付公司、AI、芯片或主流风险资产。

普通链上数据、代币行情、空投、TGE、DeFi、交易所快讯不得进入港美股日报正文。

3. 固定模板

3.1 正文结构

日报正文必须严格使用以下结构：

```markdown

港美股日报 YYYY-MM-DD

0. 今日核心结论

1. 港股打新

2. 宏观大事件

3. 产业热点与科技主线

4. 美股指数

5. 全球经济指标

6. A股 & 港股

```

3.2 模板锁定规则

禁止：

  • 改栏目顺序。
  • 改 0 至 6 的编号。
  • 删除固定栏目。
  • 新增同级大栏目。
  • 把港美股日报改成区块链日报结构。
  • 把第 2 部分写成普通 crypto 新闻池。
  • 把第 3 部分写成普通市场快讯列表。
  • 把第 4、5、6 部分改成无表格行情描述。

允许：

  • 在固定栏目内更新当天内容。
  • 在固定子模块内增加必要条目。
  • 因当天无数据而写明“本窗口无合格项目”，但不得删除栏目。

3.3 永久删除项

以下内容不得作为港美股日报正文模块出现：

  • 独立的“数据来源与可信度”大段。
  • 内部留痕、调试日志、抓取过程。
  • 被排除项目列表。
  • crypto 数据源状态列表。
  • `[A]`、`[B]`、`[C]`、`confidence A/B/C`。
  • 普通 DeFi、空投、代币行情、链上数据。

4. 时间窗口

4.1 正式日报窗口

港美股日报正式信息窗口按市场会话处理：

  • 港股与 A 股：最近一个交易日收盘至当日生成时间。
  • 美股：最近一个美股交易日收盘至当日生成时间。
  • 宏观与全球资产：最近一个可确认数据时间至当日生成时间。

如果生成日是周末或休市日，必须在标题下方写清：

```text

本日报为周末/休市日版本，行情数据截至最近交易日收盘。

```

4.2 临时截点

如果用户要求“截止现在”“截止 13 点”“按 16 点生成”，这是临时手动版本。

临时版本规则：

  • 标题或元信息写明生成时间。
  • 信息收集截止到用户指定时间。
  • 不改变正式日报的市场会话规则。
  • 如内容覆盖网站，必须说明这是当天新版本覆盖。

4.3 数据时间戳

以下内容必须带数据时间或市场会话：

  • 美股指数。
  • 全球经济指标。
  • A 股与港股指数。
  • 港股打新券商数据。
  • IPO / 融资 / ETF 状态。

不得把旧数据包装成当日实时数据。

5. 信源规则

5.1 信源优先级

优先级从高到低：

1. 官方来源：SEC、公司 IR、HKEX、Nasdaq、NYSE、Fed、BLS、BEA、Treasury、上市公司公告、上交所、深交所、JPX/TSE、KRX、Euronext、各国监管机构。

2. 原始数据源：交易所行情、Yahoo Finance、Nasdaq Data、Stooq、FRED、CBOE、HKEX、Treasury 数据。

3. 主流财经媒体：Reuters、Bloomberg、CNBC、AP、WSJ、MarketWatch、Barron's、Financial Times、Nikkei Asia、Yonhap、财联社、证券时报、格隆汇。

4. 券商与打新工具：富途、华盛、捷利、一手中签率、暗盘数据。

5. 快讯源：金十数据、财联社电报等可作为高速度触发源，但不能单独支撑高置信投资结论。

6. 二次聚合：社媒、日历工具、中文聚合资讯。

同一事实多源冲突时，优先官方与原始数据源。

产业热点与科技主线必须按全球产业链口径核查，不得只看美股新闻。系统至少覆盖中国、美国、欧洲、日本、韩国的快讯源、主流财经、官方/监管、交易所、SEC/等效披露与公司 IR；若全球源池未覆盖完整，本板块不得标记为投资级通过。

5.2 必须避免的信源错误

禁止：

  • 只看一条快讯就写成确定结论。
  • 把新闻标题改写成判断，但没有数据支撑。
  • 把过期新闻当成当日新闻。
  • 把普通 crypto 新闻放入宏观大事件。
  • 把普通裁员新闻放入 IPO / 融资，除非它与上市进程有明确关联。
  • 把 ETF 二手传闻写成已上线产品。
  • 明明有用户提供链接，却不查链接直接写缺失。

5.3 用户给定来源

用户给出链接、截图或明确来源时，必须优先处理。

处理顺序：

1. 打开或核查用户给的来源。

2. 能取到数据则更新正文。

3. 取不到则说明技术原因。

4. 再查备用源。

不得跳过用户来源。

5.4 置信度

日报正文只允许显示：

  • `置信度：高`
  • `置信度：中`
  • `置信度：低`

内部可以用 A/B/C 做筛选，但正文禁止出现：

  • `[A]`
  • `[B]`
  • `[C]`
  • `[来源][B]`
  • `confidence B`
  • `信源：...`

来源与置信度必须放在条目末尾：

```markdown

来源：SEC 8-K、公司 IR｜置信度：高

```

5.5 置信度判定

高：

  • 官方或原始数据源可确认。
  • 多个可信来源交叉验证。
  • 时间、数据、主体、市场影响清楚。

中：

  • 主流财经媒体可信，但未找到官方原文。
  • 事件可信，但细节仍需后续确认。
  • 数据来自单一原始数据源，口径清楚。

低：

  • 只有聚合源、社媒或二手线索。
  • 时间、金额、上市状态、融资细节不完整。
  • 有价值但需要人工继续核查。

低置信度内容可以进入观察表述，但不得写成确定机会。

6. 数据缺失与追查

6.1 `待补` 使用条件

只有满足以下条件之一，才允许写 `待补`：

1. 已查主数据源，未取得。

2. 已查备用数据源，仍未取得。

3. 用户提供的来源不可访问、动态加载失败或需要登录。

4. 不同来源冲突且无法判断口径。

6.2 `待补` 必须配套说明

如果字段写 `待补` 或 `口径缺失`，必须在同一表格的判断栏、备注栏或同一条内容中说明：

  • 查过哪些来源。
  • 缺失的是哪一个口径。
  • 下一步应该查什么。

禁止：

  • 明明有公开数据却写 `待补`。
  • 因为懒得核查而写 `待补`。
  • 把长段缺失说明塞进数值格。

6.3 数据缺失追查顺序

港股打新缺失时：

1. HKEX。

2. 公司招股书或公告。

3. 富途。

4. 华盛。

5. 捷利。

6. 其他券商或打新聚合。

宏观大事件缺失时：

1. 公司 IR。

2. SEC。

3. Nasdaq / NYSE。

4. Reuters / Bloomberg / CNBC / WSJ。

5. MarketWatch / Barron's / 财联社 / 格隆汇。

指数行情缺失时：

1. 交易所或官方指数源。

2. Yahoo Finance。

3. Stooq。

4. TradingView。

5. 其他可校验行情源。

宏观指标缺失时：

1. Fed / Treasury / BLS / BEA / CBOE。

2. FRED。

3. Yahoo Finance。

4. Stooq / TradingView。

7. 第 0 部分：今日核心结论

7.1 内容要求

第 0 部分写 3 至 5 条核心结论。

每条必须覆盖至少一种信息：

  • 港股打新机会强弱。
  • 宏观大事件主线。
  • IPO / 融资 / ETF 资本市场线索。
  • 美股指数与风险偏好。
  • 全球宏观变量。
  • A 股与港股市场状态。

7.2 写法要求

必须是判断句，不是标题堆砌。

正确：

```markdown

  • 周末版本行情截至最近交易日收盘，美股三大指数回落，宏观风险偏好偏谨慎，正文行情表已标注数据口径。

```

错误：

```markdown

  • 美股下跌。
  • 港股打新。
  • 今天有 IPO。

```

7.3 一致性要求

第 0 部分不得出现后文没有支撑的数据或判断。

生成后必须反查：

  • 第 0 部分提到港股打新，后文第 1 部分必须有对应项目或缺失说明。
  • 第 0 部分提到美股主线，后文第 2 部分必须有对应事件。
  • 第 0 部分提到 IPO / 融资，后文第 3 部分必须有对应条目。
  • 第 0 部分提到指数或宏观，后文第 4、5、6 部分必须有对应数据。

8. 第 1 部分：港股打新

8.1 入选标准

可入选：

  • HKEX 新上市申请、聆讯后资料集、招股书、公告。
  • 正在招股、即将上市、配发结果、暗盘表现。
  • 券商孖展、入场费、一手中签率、基石投资者、超购倍数。

不应入选：

  • 只有传闻、无 HKEX 或公司文件支撑的项目。
  • 与港股 IPO 无关的公司新闻。
  • 已过期且无当日更新价值的旧招股信息。

8.2 主页面呈现规则

港股打新仍然是主日报第 1 部分，不得删除或弱化为纯跳转入口。

主页面第 1 部分必须保留，但版面要压缩：只呈现综合评述、关键项目摘要、核心风险变量和子网站链接，不在主页面展开全部项目明细。

原因：

  • 港股打新很多时候受美股风险偏好、全球宏观、港股科技板块情绪影响。
  • 主日报阅读顺序应先看宏观大事件、全球宏观和 A/H 股状态，再回到港股打新子页面做细节判断。
  • 主页面负责给出第一层判断，让读者知道今天港股打新的整体价值、主要风险和是否值得点进子页面；子页面负责承载全部申购细节。

主页面固定格式：

```markdown

港股打新综合评述

综合判断：...

今日重点：1-3 个项目或线索摘要，不展开全量字段。

风险变量：...

细节入口：{港股打新子网站链接}

来源：HKEX、券商数据｜置信度：高/中/低

```

主页面不得出现完整项目列表、全量招股书清单、所有券商字段或长表格；但必须保留第 1 部分的独立内容和判断，不能只放一个链接。

8.3 子网站明细规则

港股打新明细放在子网站或子页面中。子页面可以展示完整项目表。

子页面固定表格为：

公司代码/状态阶段关键数据判断

字段要求：

  • 公司：公司全称，可带简称。
  • 代码/状态：已分配代码则写代码；未分配则写 `未定` 或 `聆讯/递表`。
  • 阶段：递表、聆讯后、招股中、定价、配发结果、暗盘、上市。
  • 关键数据：入场费、孖展、一手中签率、基石、招股日期、上市日期等，数值格保持简洁。
  • 判断：是否值得跟踪、缺什么数据、下一步看什么。

8.4 子页面条目格式

子页面如果不用表格，每个项目必须包含：

```markdown

公司名称

阶段：...

关键数据：...

判断：...

来源：...｜置信度：高/中/低

```

8.5 主页面判断要求

主页面综合判断必须回答：

  • 港股打新当前是否只是名单跟踪，还是值得进入子页面做细节判断。
  • 美股风险偏好和全球宏观是否支持新股情绪。
  • 港股科技或恒生科技表现是否影响打新风险偏好。
  • 哪些字段仍缺失，是否影响申购判断。
  • 为什么读者应该现在点进子页面，或为什么暂时只观察。

不得只写：

  • “今日有若干项目。”
  • “详情见链接。”
  • “后续观察。”

8.6 子页面判断要求

判断必须回答：

  • 当前是否只是名单跟踪，还是可以形成申购判断。
  • 是否有券商热度、孖展、一手中签率支持。
  • 后续看什么变量。

不得只写：

  • “值得关注。”
  • “后续观察。”
  • “资料待补。”

8.7 缺失处理

缺入场费、孖展、一手中签率、暗盘、基石投资者时，不得删除项目。

主页面必须写清缺失是否影响综合判断；子页面必须在判断栏写清：

```markdown

已查 HKEX 招股书；券商孖展、一手中签率和暗盘热度待补，下一步查富途、华盛、捷利。

```

9. 第 2 部分：宏观大事件

9.1 入选标准

可入选：

  • Fed、CPI、PCE、非农、债券收益率、美元、油价、黄金、VIX 等影响全球风险偏好的事件。
  • 地缘政治、战争、制裁、能源供给、航运通道等会影响油价、美元、美债、风险资产的事件。
  • 美国、欧洲、中国、日本、韩国等主要市场的高影响政策、监管、央行和宏观事件。
  • 对上市公司、行业板块、ETF、IPO、交易所产品有明确影响的监管或资本市场事件。
  • 对上市公司、ETF、IPO、交易所产品有明确影响的加密资本市场事件。

不应入选：

  • 普通 crypto 市场快讯。
  • 无上市公司或市场影响的视频标题。
  • 只有二手快讯、无主体和时间的传闻。
  • 单纯个股涨跌且无事件驱动。
  • 小公司低影响新闻，除非属于 IPO、融资或监管重点。

9.2 固定条目格式

每条必须包含：

```markdown

事件标题

事件：...

判断：...

来源：...｜置信度：高/中/低

```

格式必须高度压缩，优先结论和交易变量，不展开长段背景。

9.3 判断要求

市场判断必须回答：

  • 对全球风险偏好、利率、油价、美元、美股、港股或 A/H 科技资产的影响方向。
  • 影响是短期、阶段性还是结构性。
  • 后续看什么变量，以及什么情况会推翻当前判断。

不得只写：

  • “值得关注。”
  • “影响较大。”
  • “后续观察。”

9.4 crypto 例外规则

区块链相关内容只有满足以下条件之一，才可进入本栏目：

  • 涉及 Coinbase、Robinhood、Circle、Gemini、矿企、加密财库公司等上市公司。
  • 涉及 Kraken、Ledger 等明确 IPO / 上市进程。
  • 涉及 BTC / ETH / crypto ETF 的 SEC、交易所或发行方事件。
  • 涉及上市公司持币、财报、并购、监管处罚。
  • 对美股金融机构、支付公司、券商、交易所产生明确影响。

普通 DeFi、TGE、空投、代币涨跌不得进入。

10. 第 3 部分：产业热点与科技主线

10.1 板块定义

本板块服务于每日可影响投资判断的产业主线，不再以 IPO / 融资为固定主轴。

可包含：

  • AI、半导体、算力、存储、云、机器人、软件等科技产业主线。
  • 重点公司财报、业绩预告、指引、订单、资本开支、库存和价格周期。
  • A/H 产业链事件对美股科技股、全球半导体链或港股科技情绪的映射。
  • 中美欧日韩重要产业链新闻、公司公告、监管/交易所披露，对港美股和 A/H 科技资产有明确映射的事件。
  • 美股 IPO、S-1、F-1、424B、上市、SPAC。
  • 港股 IPO 与第 1 部分不同角度的重点上市事件。
  • 重大科技融资。
  • 重大并购。
  • 上市公司战略投资。
  • ETF、基金产品或交易所新产品。

不可包含：

  • 普通市场新闻。
  • 纯价格新闻。
  • 没有上市、融资、并购、基金产品实质的宣传稿。
  • 普通裁员新闻，除非与 IPO、上市重组或资本市场交易直接相关。

10.2 固定子模块

第 3 部分固定包含：

1. 今日产业主线判断。

2. AI / 半导体 / 算力链。

3. 重点公司财报 / 指引 / 订单 / 资本开支。

4. IPO / 融资 / 资本市场事件。

如果某子模块无合格内容，写明确缺口，不使用旧式空占位。

```markdown

本窗口未出现足够影响产业判断的新增线索。

```

不得删除子模块。

10.2.1 全球产业源池要求

第 3 部分采集不得局限于美股新闻。系统必须同时维护：

  • 中国信源：财联社、金十数据、证券时报、上交所、深交所、HKEX、中国证监会。
  • 美国信源：Reuters、AP、CNBC、Bloomberg、SEC EDGAR、Nasdaq、NYSE、公司 IR。
  • 欧洲信源：Financial Times、Euronext、ASML IR 及关键监管/产业政策来源。
  • 日本信源：Nikkei Asia、JPX/TSE、Tokyo Electron IR。
  • 韩国信源：Yonhap、Korea Exchange、Samsung Electronics IR、SK hynix IR。

重点公司池至少覆盖 AI GPU、云/AI Capex、半导体设备、晶圆代工、DRAM/HBM、AI 服务器、机器人/自动驾驶等链条。快讯源只负责发现线索，最终判断必须尽量回填官方/公司 IR、交易所、SEC 或主流财经交叉确认。

10.3 IPO 项目固定字段

每个 IPO / 上市项目必须包含：

```markdown

公司名称 / 产品名称

类型：IPO / F-1 / S-1 / 上市 / SPAC / ETF

公司说明：...

资本市场进展：...

关键数据：...

判断：...

下一步：...

来源：...｜置信度：高/中/低

```

关键数据可包含：

  • 交易所。
  • 股票代码。
  • 融资规模。
  • 估值。
  • 发行价区间。
  • 上市日期。
  • 承销商。
  • 文件状态。

缺失时写 `待补`，并在判断或下一步说明追查来源。

10.4 融资事件字段

当产业热点涉及融资事件时，必须包含：

```markdown

公司名称

公司说明：...

融资：...

投资方：...

市场相关性：...

下一步：...

来源：...｜置信度：高/中/低

```

融资字段规则：

  • 有公开融资金额：写金额。
  • 无公开金额：写 `未披露`。
  • 不确定：写 `待核查`，并说明需要查的来源。

投资方规则：

  • 有明确投资机构：写机构名称。
  • 未披露：写 `未披露`。
  • 只有生态支持、合作伙伴、客户合同：不得写成投资方。

10.5 ETF / 基金产品事件字段

当产业热点涉及 ETF / 基金产品，且满足重大市场影响和官方确认要求时，必须包含：

```markdown

产品名称

发行方：...

标的：...

交易所/代码：...

状态：...

判断：...

来源：...｜置信度：高/中/低

```

状态必须区分：

  • 已上市。
  • 已提交申请。
  • 已获批待上市。
  • 传闻 / 待核查。

不得把“分析师称可能上线”写成“已上线”。

11. 第 4 部分：美股指数

11.1 固定表格

指数数值涨跌判断
S&P 500数字涨跌判断
Nasdaq Composite数字涨跌判断
Dow Jones数字涨跌判断

可按需要增加：

  • Russell 2000。
  • Philadelphia Semiconductor Index / SOX。
  • Magnificent 7 / 等权重指数。

不得删除三大指数，除非数据源完全不可得并写明追查。

11.2 数值格硬规则

`数值` 和 `涨跌` 两列只写数字、百分比或极短标签。

允许：

  • `7,408`
  • `-0.49%`
  • `待补`
  • `口径缺失`

禁止：

  • 来源名。
  • 日期。
  • 长句。
  • API 状态。
  • 判断句。

11.3 判断栏要求

判断栏必须写清：

  • 数据日期或市场会话。
  • 指数趋势。
  • 对风险偏好的含义。
  • 来源差异或口径问题。

正确：

```markdown

截至 5月15日美股收盘，科技权重回落，短线风险偏好偏谨慎；来源 Stooq，后续需用 Yahoo Finance 交叉校验。

```

错误：

```markdown

待补，之后再看。

```

11.4 表后简评

表格下方必须有 1 段简评。

简评必须回答：

  • 三大指数整体偏强还是偏弱。
  • 科技股相对强弱。
  • 是否存在宏观或利率变量影响。

14. 网页生成规则

14.1 HTML 要求

网页必须体现固定正文结构。

表格要求：

  • 边框清楚。
  • 行列对齐。
  • 移动端可读。
  • 不因长文本撑破页面。
  • 数值格保持简洁。
  • 判断栏允许较长，但不得无限撑宽。

14.2 表格宽度规则

桌面端：

  • 表格宽度使用正文可用宽度。
  • 不得窄于正文主体宽度的 90%。
  • 判断栏可以更宽，数值列保持紧凑。

手机端：

  • 表格允许横向滚动或自适应换行。
  • 不得让页面整体横向溢出。
  • 长判断必须换行。

14.3 CSS 缓存

修改 `css/style.css` 后，所有引用该 CSS 的日报 HTML 必须更新版本号。

示例：

```html

<link rel="stylesheet" href="/daily-brief/css/style.css?v=hk-us-v1">

```

版本号要随实际修改变化，不得沿用旧缓存参数。

14.4 预览刷新

线上预览时建议使用刷新参数：

```text

?refresh=时间戳

```

如果用户反馈页面没变，先排查缓存，再判断是否发布失败。

16. 飞书推送规则

16.1 每日固定自动化规则

港美股日报必须设置每日固定自动化：

  • 执行时间：每天 09:00 CST。
  • 任务目标：生成当日港美股日报，发布到 GitHub Pages 网页，并完成飞书推送。
  • 发布顺序：先生成日报，再通过质量门禁，再发布网页，再验证网页链接，最后推送飞书。
  • 飞书推送必须在网页链接可验证后执行；网页未发布或未验证时，不得推送“已生成”。
  • 飞书推送内容只能包含一句话和网页链接，不得推送日报全文、内部日志、质量评分、数据源缺口、测试输出或调试信息。
  • 如果自动化当天失败，必须明确记录失败阶段：生成失败、质量门禁失败、网页发布失败、网页验证失败、飞书推送失败。

每日正式推送文案：

```text

今日港美股日报已生成：{网站地址}

```

其中 `{网站地址}` 必须使用可打开的完整 GitHub Pages URL，不得只使用本地路径或相对路径。

港美股项目中的 `notify` 命令如果只是生成通知文本文件，不等于飞书已推送。只有真实调用飞书发送接口并成功返回，才可以记为“飞书已推送”。

正式自动化必须使用真实飞书发送路径：

```bash

PYTHONPATH=src python3 -m hk_us_daily.cli check-feishu

PYTHONPATH=src python3 -m hk_us_daily.cli notify "{完整网站地址}" outputs/YYYY-MM-DD-notification.txt --date YYYY-MM-DD --send-feishu

```

飞书推送所需配置：

```text

FEISHU_APP_ID

FEISHU_APP_SECRET

FEISHU_HK_US_CHAT_ID

FEISHU_HK_US_RECEIVE_ID_TYPE=chat_id

```

配置可以来自环境变量，也可以来自港美股日报工作区 `.env` 文件。缺少任一必要配置时，`check-feishu` 必须失败；不得降级为只写通知文本后声称飞书已推送。

如果只完成网站未完成飞书，必须说明“网站已发布，飞书未推送”。

如果飞书推送失败，必须说明失败原因和下一步。

不得把“网站已发布”说成“全流程完成”。

25. 漂移判定规则

只要出现以下任一情况，即判定为模板漂移。

25.1 结构漂移

  • 0 至 6 部分顺序变化。
  • 新增第 7 部分或其他同级大栏目。
  • 删除任一固定栏目。
  • 把固定表格改成段落。
  • 把固定栏目改名。
  • 把港美股日报改成区块链日报栏目。

25.2 内容字段漂移

  • 港股打新在主页面展开完整项目表。
  • 港股打新第 1 部分被删除，或只剩纯链接没有综合判断。
  • 港股打新主页面缺综合判断、今日重点、风险变量、细节入口、来源与置信度任一项。
  • 港股打新子页面缺公司、阶段、关键数据、判断、来源与置信度。
  • 宏观大事件缺事件、判断、来源或置信度。
  • IPO / 融资 / ETF 缺固定字段。
  • 美股指数缺 S&P 500、Nasdaq Composite、Dow Jones 任一项。
  • 全球经济指标缺黄金、原油、美元指数、10Y 美债、VIX 任一项。
  • A 股 & 港股缺上证、深证、恒生、恒生科技任一项。

25.3 表达漂移

  • 使用英文 A/B/C 置信度。
  • 把来源放到条目前面。
  • 把来源集中放到文末。
  • 用“信源”作为正文标签。
  • 把内部留痕写进正文。
  • 用“排除项”形式解释为什么某内容没入选。

25.4 数据处理漂移

  • 明明有数据源却写 `待补`。
  • `待补` 没有追查说明。
  • 数值格塞入日期、来源、长句。
  • 口径冲突没有写进判断栏。
  • 把休市日旧行情写成当日实时行情。

25.5 网页样式漂移

  • 表格宽度明显窄于正文。
  • 手机端表格横向撑破页面。
  • 数值列和判断列混乱。
  • 调试信息出现在网页正文。
